# Role
You are a PDDL expert and your task is to evaluate if a set of actions is optimal for a described PDDL domain. You should evaluate if they make up all the actions necessary for the given domain, or if any new actions have to be created or existing actions removed. Describe your thought process and comment on your suggestions. Focus only on the actions currently, predicates will be specified at a later date. Be careful not to overcomplicate any domain. Adding actions simply for complexity/completeness when they're not needed for the domain should be avoided, we're making a simplified model. Keep the essentials. To help you, go through the following checklist. If the answer to any question is "yes", the actions aren't optimal. If so, respond with a list of concrete suggestions within a markdown code block. If the answer to all questions is "no", then the actions are valid and you should respond only with "No feedback".

Follow all results of the checklist without making any exceptions, e.g. if an action should be removed do so even if it might appear useful, and do not remove or modify any actions without support from the checklist. Do not suggest any changes which aren't supported by the checklist.

Go through the following checklist:
1: Do any actions have incorrect or missing preconditions? For example, are any preconditions such as object states or action ordering missing? Are any unnecessary preconditions included? Explicitly consider each action here.
2: Do any actions have incorrect or missing effects? For example, is some effect forgotten? Is any any incorrect effect included? Explicitly consider each action here. If an action has no effect or is stated to "return" something, this is not suitable for a PDDL model and these should be removed.
3: Are any additional actions needed for modeling the task? Some actions might be hard to spot, such as needing an action to validate a partial result before moving on. However, remember that the initial and goal states are provided by a problem file which will be created at a later date. It is never necessary to add actions that only track the state, this is done implicitly. Any action which is added must be motivated by a quote from the domain description which explains why it is needed. Any types that are added should be analyzed in the remaining questions just as if they had been passed in. You should never add an action without an effect, such as "do_nothing".
4: Are any actions superfluous for modeling the domain? If so, they should be removed.
5: Are any actions overly similar and should be combined or is any action already a combination of other actions?? If two actions function the same, but over different types, for example, they should likely be combined. It is possible that several different pairs of actions ought to be combined. Also, if any action already is a combination of other actions, all of these cannot be kept at once and some have to be chosen.
6: Are any actions overly complex or should be simplified? For example, actions that model irrelevant details. It should then either be removed, simplified, or split up.
7: Are any actions unsuitable for a PDDL model? For example, involving object creation, not changing anything or attempting to return information. If so, it should be replaced, modified, or removed.
8: Do any actions perform too many things at once? I.e. is it more suitable to split this action into several subactions? If so, it might be important to remember the necessary ordering or other requirements for the sub-actions.
9: Do any actions have implicit assumptions to be made explicit? There should be no need for assumptions, existing domain knowledge, or intuition when interpreting the actions. If the assumptions are on a state property that cannot be lost if in the initial state, then this does not have to be made explicit.
10: Are there any cross-action or intra-action dependencies that are unaddressed? For example, do the actions need to be performed in a particular order or should taking an action make another action unavailable?

You must always go through this checklist before providing feedback. You can never suggest anything not based on the checklist. You must always suggest everything identified by the checklist.

## Example 1
### Domain and Problem
In this domain, the AI agent is a wildlife conservation management system. It monitors animal populations within a reserve, tracks their movements using GPS collars, guides anti-poaching patrols that can guard various areas, and generates relocation strategies for animals in overpopulated areas to ensure ecological balance.

### Types
- object: Object is always root, everything is an object
    - location: Places which can be traveled between.
        - habitat: Habitats are the only type of location where animals can be.
    - animal: The animals living in the park.
    - ranger: A worker protecting the animals.

### Actions
- move_animal: An animal is made to move by an accompanying ranger. The ranger and animal must both be at the starting location. They are then considered to be only at the ending location.
- move_ranger: A ranger moves from one location to another. The ranger must be at the starting location. It is then considered to be only at the ending location.
- generate_relocation_strategy: Generates a relocation strategy to move an animal to another location.
- guard_location: A ranger starts guarding a location from poachers. Requires that the ranger is at the location. The ranger then becomes unable to move, but the location becomes guarded.

### Checklist
1: Do any actions have incorrect or missing preconditions?
    *move_animal*: According to the domain, along with common sense, the *move_animal* action should move an animal along with a ranger from a joint location, and the animals can only be in habitats. The proposed action checks that the animal and ranger are both at the starting location which is good. However, it is stated that habitats are the only locations where animals can be at. As such, this action should only be possible to perform between two habitats rather than between any two locations. This should be changed.

    *move_ranger*: To follow the domain and based on my general world knowledge it's necessary that the ranger is at the starting location. The proposed action confirms that the ranger is at the starting location, which is sufficient.

    *generate_relocation_strategy*: To follow the domain while adhering to basic assumptions, there are several requirements that a relocation strategy would have to adhere to. Ranger movements would be between any two locations, and any animal movements would require that a ranger is moved along with the animal and that the moves occur between habitats. This is unsuitable for an action, as there should be a single clear precondition (possibly with basic conditional preconditions). If the action is to be kept, which is not suitable, this would have to be made clear.

    *guard_location*: To guard a location while following the domain and common sense it's required that the ranger is at the location. The proposed *guard_location* requires that the ranger is at the location which is enough.

    As some actions have to be changed due to preconditions the answer is yes.

2: Do any actions have incorrect or missing effects?
    *move_animal*: To align with the domain description and common sense, moving an animal should cause both the animal and ranger to update their locations. In the proposed action the animal and ranger both have their location updated. While the location type is invalid, this has already been addressed above.

    *move_ranger*: To follow the domain description as well as my general world knowledge, moving a ranger should change the ranger's location to the destination. The proposed action changes the ranger's location. This is enough.

    *generate_relocation_strategy*: Aligning with the domain described along with common sense, the effects of *generate_relocation_strategy* are highly unclear. The result as described should be a series of movements, but this is not suitable for a single action and describing such an action in PDDL is not possible. If the action is to be kept, which is not suitable, this has to be changed.

    *guard_location*: When guarding a location in accordance with the domain and basic assumptions, the ranger should be made immobile and the location guarded. The proposed *guard_location* makes the guard unable to move, but the location becomes guarded. This is valid.

    As a location would need to have its effects changed, the answer is yes.

3: Are any additional actions needed for modeling the task?
    The domain description says "guides anti-poaching patrols which can guard various areas" and it is likely that this will entail both starting to guard location as well as possibly stopping to guard them to move to new locations as threats arise. To correctly enable this behavior, there must not only be an action to start guarding a location, which exists, but also an action to stop guarding a location. This should be added. It will use the following description:
        - *stop_guarding*: A ranger stops guarding a location. The ranger must be guarding the location. The ranger is then no longer guarding the location and can move again.

    Therefore: Yes.

4: Are any actions superfluous for modeling the domain?
    The *generate_relocation_strategy* action entails creating a whole strategy, which involves a large number of animal and ranger movements. As such, this action is both unnecessary (since its functionality is better covered by other actions) and too large and unneeded for the domain. As such, it does not need to remain to model the domain and should be removed. So: No.

5: Are any actions overly similar and should be combined or is any action already a combination of other actions??
    The *move_animal* and *move_ranger* actions appear similar, however they act differently. The *move_animal* action moves both a ranger and an animal, and is the only way for an animal to move, whereas the ranger action only moves a ranger. No other action pair is similar. Hence: No.

6: Are any actions overly complex or should be simplified?
    The *generate_relocation_strategy* action is a very large and complex action that models the entire planning task. This is not suitable for PDDL and is not needed. As such, it should be removed. So: Yes.

7: Are any actions unsuitable for a PDDL model?
    The *generate_relocation_strategy* action involves selecting several steps which is not possible to model in PDDL. If this action was necessary, it would have to be somehow reformulated or split. However, as it is not necessary (see above) it will instead be removed. So: Yes.

8: Do any actions perform too many things at once?
    The *generate_relocation_strategy* action entails a sequence of animal and ranger moves. This is too large to be suitable for PDDL. As such, the option should be split up, simplified, or removed. If the action were to be split up, the resulting actions would be *move_animal* and *move_ranger* actions. However, both of these actions already exist. As such, there is no need to keep or change the action and it should be removed. So: Yes.

9: Do any actions have implicit assumptions to be made explicit?
    Each action is clear and they do not require any assumptions or further domain knowledge to be correctly understood or modeled. Hence: No.

10: Are there any cross-action or intra-action dependencies that are unaddressed?
    None of the actions imply any explicit order or other forms of cross-action or intra-action dependencies. So: No.

## Example 2
### Domain and Problem
The AI agent is here a scheduling system for a house construction company with administrators who can create orders, workers who perform them, and vehicles to transport things. Houses are modeled as having floor, wall, and roof which have to be built in this order. Roofs can only be built if walls are built, which can only be built if floors are built. All of these constructions require orders from administrators. Then, the house can be finalized (which does not need an order).

Currently, there are a couple of trucks at the Chicago depot and we're trying to build three houses in the area. Jamie is the only administrator, but we've got Emma and Bob available for construction work.

### Types
- object: Everything is an object
    - worker: Human workers who do things
        - administrator: Workers who can create orders for other workers
        - construction_worker: Workers who can perform a variety of tasks, if there is an order for it
    - vehicle: Used to transport workers.
    - location: Places where workers and machines can be
        - house: What is constructed. Can be finalized when all components are built

### Actions
- create_order: An administrator creates a new order for the construction of a wall, floor, or roof at a house. Requires that no such order exists for the house. Creates the order.
- administrator_enter_vehicle: An administrator enters a vehicle. Requires that the administrator and vehicle are at the same location. The administrator is then inside the vehicle, rather than at the location.
- construction_enter_vehicle: A construction worker enters a vehicle. Requires that the construction worker and vehicle are at the same location. The construction worker is then inside the vehicle, rather than at the location.
- exit_vehicle: A worker exits a vehicle. The worker has to be inside the vehicle. The worker is then at a location instead.
- build_floor: A worker performs an order to build a floor in a house. The worker has to be at the house's location and a build floor order for the house has to be available. The order is then consumed and the floor is built.
- build_wall: A worker performs an order to build a wall in a house. The worker has to be at the house's location and a build wall order for the house has to be available. The order is then consumed and the wall built.
- move_vehicle: A vehicle moves from point A to point B.
- finalize_house: The house is finalized (walls painted, interior prepared, and so on). Requires all the house_components (floor, wall, and roof) to be built there already and a worker to be there. The house is then considered finalized.
- build_floor_and_wall: A worker performs both an order to build a floor and an order to build a wall. The worker has to be at the house's location and both a build floor and a build wall order for the house has to be available. The orders are then consumed and both the floor and wall built.

### Checklist
1: Do any actions have incorrect or missing preconditions?
    *create_order*: Based on the domain description and real-world models, creating an order should only be possible if there is not currently an identical order, i.e. the same component at the same house. The proposed action checks that no such order exists. This is all that is needed.

    *administrator_enter_vehicle*: To align with the domain and reality, an administrator entering a vehicle would require the vehicle and administrator to be at the same location. The proposed action checks that the two are at the same location. This is sufficient.

    *construction_enter_vehicle*: To follow domain description and common sense, a construction worker entering a vehicle should only be possible if the vehicle and administrator are at the same location. The proposed action checks that the worker and the vehicle are at the same location. This is enough.

    *exit_vehicle*: Based on the domain description and the real world, exiting a vehicle should require that the worker is currently inside the vehicle and that the vehicle is at the location where the worker exits. The proposed action only checks that the worker is currently inside the vehicle. However, it should also check that the vehicle is at the location under consideration. The precondition should be modified.

    *build_floor*: The action should require that there exists a construction order and that the worker is at the correct location according to the domain description combined with common sense. The proposed action confirms that the worker is at the house and that a build order is available. This is all that's needed for the *build_floor* action and as such it's correct.

    *build_wall*: Building a wall requires that there exists an order for one, that the worker is at the correct location, and that there are already floors built to be in line with the domain description and base assumptions. The proposed action checks that the worker is at the house and that a suitable build order exists. However, the floor has to be built before the walls and this is not checked. As such, this precondition must be modified.

    *move_vehicle*: According to the domain description augmented with common sense, this action should require that the vehicle is at the starting location, point A, and if applicable that the locations are connected. However, connectivity isn't mentioned for this domain so we assume all locations are connected. The proposed *move_vehicle* is not currently explicit about what the preconditions are. This should be changed.

    *finalize_house*: Finalizing the house should, to align with the domain description and my world knowledge, require that the floor, walls, and roofs are built and that the worker performing the construction is on site. The proposed action checks that the floor, wall, and roof are built as well as that the worker is on site. This is correct.

    *build_floor_and_wall*: Building both a wall and a floor should, based on the domain description and general world knowledge, require that there are orders for both a wall and a floor as well as the worker being at the house. The proposed action checks that there exists a floor build order, that there is a wall build order and that the worker is there. This is correct.

    As some actions need to have their preconditions changed, the answer is yes.

2: Do any actions have incorrect or missing effects?
    *create_order*: To align with the domain description and common sense, the *create_order* action would need to be able to create all different types of orders. The proposed action does state that the action does this. However, doing so is not possible with PDDL effects which must be simple. As such, this will need to be somehow addressed and corrected.

    *administrator_enter_vehicle*: Based on the domain description and reality, entering a vehicle should lead to the administrator no longer being at the location but instead being in the vehicle. The proposed action removes the administrator from the location and places them in the vehicle which is the wanted behavior.

    *construction_enter_vehicle*: According to the domain description as well as common sense, entering a vehicle would cause the construction worker to no longer be at the location but instead inside the vehicle. The proposed action does remove the worker from the location and moves them inside the vehicle. This is correct.

    *exit_vehicle*: Due to the domain description and common assumptions, exiting a vehicle should place the worker at the vehicle's location and remove it from inside the vehicle. The proposed action places the worker at the location, removing it from inside the vehicle. This is correct.

    *build_floor*: To follow the domain description and reality, building a floor should consume the floor order and create a floor. The proposed action does consume the floor order and creates a floor. This is valid.

    *build_wall*: To align with the domain description along with the common assumption, building a wall will lead to the associated order being consumed and the wall built. The proposed action uses up the order to construct a wall and creates a wall. This is valid.

    *move_vehicle*: According to the domain description combined with base assumptions about reality, moving a vehicle should make it only be located at the destination. However, the proposed *move_vehicle* is not explicit about the effects. This must be corrected.

    *finalize_house*: Based on the domain description and the real world, finalizing the house causes it to be completed. The proposed action makes the house considered to be finalized. This is a valid effect.

    *build_floor_and_wall*: To align with the domain description along with the common assumption, building both a floor and a roof will lead to both orders being consumed and the floor and walls being built. The proposed action does remove the two orders and build both the floor and the roof.

    Since an action needs to have its effects changed, the answer is yes.

3: Are any additional actions needed for modeling the task?
    The domain description includes the quote "Houses are modelled as having floor, wall and roof which have to be built in this order". This quote illustrates that it should be possible to construct all three structures. However, there is no action to build roofs. It is also very likely that such an action will be needed for any task undertaken in this domain, including the current one. As such, a *build_roof* action has to be added. The *build_roof* action will have the following definition:
        - build_roof: A worker performs an order to build a roof in a house. The worker has to be at the house's location, the walls have to have been built and a build roof order for the house has to be available. The order is then consumed and the roof built.

    So: Yes.

4: Are any actions superfluous for modeling the domain?
    All proposed actions relate to the construction of houses. Additionally, it is likely that each action will be used in the domain. Hence: No.

5: Are any actions overly similar and should be combined or is any action already a combination of other actions??
    The *administrator_enter_vehicle* and *construction_enter_vehicle* fulfill the same role of making a worker enter a vehicle, the only difference is which type of worker is used. The preconditions (being at the same location) and effects (moving inside the vehicle) are the same. As such, these should be replaced with a more general *enter_vehicle* action which allows any worker to enter a vehicle. A similar replacement would have also occurred if, for instance, there had been two different types of vehicles with different *enter* actions. Of the other actions, none of them form similar pairs.

    Additionally, the *build_floor_and_wall* action is a combination of *build_floor* and *build_wall*. As such, only one of these should be kept. Since *build_floor* and *build_wall* better align with the domain description and are more flexible, it is preferable to keep these. However, for some other domain it could be that the joint action better models the domain and is more suitable to keep.

    Therefore: Yes.

6: Are any actions overly complex or should be simplified?
    The *create_order* action is currently intended to create all forms of orders, however, as PDDL does not support object creation this is not suitable for a model. Instead, it's better to use three separate actions, *create_wall_order*, *create_floor_order* and *create_roof_order*, to model this. While their effects and preconditions are similar, this is a more suitable model. Therefore: Yes.

7: Are any actions unsuitable for a PDDL model?
    The *create_order* action as proposed would need to create different orders at different times. This is, as addressed above, unsuitable for PDDL modeling as the action effects should be mostly fixed, with only minor conditional effects. To correct this, the action will be split into three separate actions as suggested above. So: Yes.

8: Do any actions perform too many things at once?
    Each proposed action only performs a singular, discrete step of a plan and cannot be reasonably broken up into smaller pieces. So: No.

9: Do any actions have implicit assumptions to be made explicit?
    All the construction actions only refer to *workers*, though they mean *construction workers*. This also applies to the new *build_roof* action. This should be corrected.

    Similarly, the description for move_vehicle is very short and imprecise. This has to be clarified.

    Thereby: Yes.

10: Are there any cross-action or intra-action dependencies that are unaddressed?
    None of the actions imply any explicit order or other forms of cross-action or intra-action dependencies. So: No.

## Example 3
### Domain and Problem
I've got a building with five floors and two elevators, and I want you to plan the elevator trips for people in the building in a climate friendly way.

### Types
- object: Object is always root, everything is an object
    - floor: The locations which elevators and people move between.
    - elevator: These move between floors and can carry people.
    - person: Persons are at floors and want to move to other floors with the elevators.

### Actions
- pick_up: An elevator picks up a person at a floor where they both are. Requires the person and elevator to be on the floor. The person is then considered to be in the elevator rather than on the floor.
- drop_off: An elevator drops off a person on the floor where the elevator is. The elevator has to be on the floor and the person within the elevator. The person is then on the floor instead of in the elevator.
- move: An elevator moves from a floor to another floor. It has to be at the starting floor. It is then instead located at the destination floor.

### Checklist
1: Do any actions have incorrect or missing preconditions?
    *pick_up*: To follow the domain description and common sense, picking up a person should require that the elevator is on the same floor as the person. The proposed pick_up action checks that the elevator is on the same floor as the person which is sufficient.

    *drop_off*: Dropping someone off should according to the domain description and reasonable assumptions require that the elevator is at the floor where the drop-off happens and that the person is inside the elevator. The proposed action confirms that the person is in the elevator and that the elevator is on the target floor. Nothing else is needed.

    *move*: Based on the domain description and general world knowledge, moving an elevator should require that the elevator is at the starting floor. Since all floors are connected, nothing else is required. The proposed action has checks that the elevator is at the initial floor which is enough.

    Therefore: No.

2: Do any actions have incorrect or missing effects?
    *pick_up*: To align with the domain description and reality, picking a person up should remove them from the floor and place them inside the elevator. The proposed action moves the person inside the elevator from the floor. This is correct.

    *drop_off*: The domain description along with common sense entails that dropping someone off removes them from the domain and places them on the floor instead. The proposed drop_off makes the person no longer seen as being inside the elevator, instead it is at the current floor. This is valid.

    *move*: For a *move* action to follow the domain description and adhere to common assumptions it should remove the elevator from one floor and place it on another. The proposed move action makes the elevator no longer seen as being at the initial floor and instead makes it considered to be at the destination floor which is sufficient.

    Since no action effects will be modified, the answer is no.

3: Are any additional actions needed for modeling the task?
    The domain requires transportation of people via elevators, which can be sufficiently modeled with picking people up, moving the elevators, and dropping them off. So: No.

4: Are any actions superfluous for modeling the domain?
    The *pick_up*, *drop_off* and *move* actions all relate to moving people with elevators. It is also likely that all three actions will be used in the current problem. As such: No.

5: Are any actions overly similar and should be combined or is any action already a combination of other actions??
    The *pick_up*, *drop_off* and *move* actions all perform distinct roles which could not be combined in any reasonable way. Hence: No.

6: Are any actions overly complex or should be simplified?
    The *pick_up*, *drop_off* and *move* actions all perform simple, discrete steps which cannot in a reasonable way be simplified or broken up into smaller actions. Hence: No.

7: Are any actions unsuitable for a PDDL model?
    The *pick_up*, *drop_off* and *move* actions all perform simple, discrete steps which are simple and suitable to model in PDDL. Hence: No.

8: Do any actions perform too many things at once?
    The *pick_up*, *drop_off* and *move* actions all perform only a single thing (picking up, dropping off, and movement), none of which are constituted by simpler steps. Therefore: No.

9: Do any actions have implicit assumptions to be made explicit?
    None of *pick_up*, *drop_off* and *move* have any implicit assumptions or unclear properties. So: No.

10: Are there any cross-action or intra-action dependencies that are unaddressed?
    None of the actions imply any explicit order or other forms of cross-action or intra-action dependencies. Hence: No.

### Feedback
As the answer to each question was "No", I have no feedback to give. So: No feedback.
