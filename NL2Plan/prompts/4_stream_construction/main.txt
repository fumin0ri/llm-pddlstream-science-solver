# Role
You are a PDDL expert, and your task is to define the preconditions and effects (represented in PDDL format) of an AI agent's actions. Information about the AI agent will be provided in the domain description. Remember the following:
    - Individual conditions in preconditions and effects should be listed separately. For example, "object_1 is washed and heated" should be considered as two separate conditions "object_1 is washed" and "object_1 is heated".
    - In PDDL, two predicates cannot have the same name even if they have different parameters. Each predicate in PDDL must have a unique name, and its parameters must be explicitly defined in the predicate definition. It is recommended to define predicate names in a simple, intuitive, and readable way as well as to avoid symmetrical constraints, so (same_room ?b1 - block ?b2 - block2 ?r - room) should instead be modeled with two predicates, (in_room ?b1 - block ?r - room) and (in_room ?b2 - block ?r - room).
    - Creating new actions is not an option.
    - Avoid using pure "object" types as far as possible for action parameters, they should only be used if absolutely necessary. The supplied set of types should be sufficient to model the problem, along with grounded predicates.
    - The domain description always takes precedence over the action description. The action description might be wrong, or might have forgotten something but the domain description is always right.

Start by defining the action parameters under the "### Action Parameters" heading with each parameter on a new line of a markdown code block following the "- {{parameter name}} - {{parameter type}}: {{parameter description}}" format. Then, analyze the preconditions under the "### Action Preconditions" heading, finished with a PDDL precondition within a markdown block. Then, describe the action effects under the "### Action Effects" heading, ending with a PDDL effect within a markdown block. Lastly, define the newly created predicates under the "### New Predicates". Each predicate should be on its own line within a markdown code block, following the "- ({predicate name} {predicate parameters}): {predicate description}" format. It is good praxis to always start each action precondition and effect with an enveloping "and" statement.

When defining new predicates, you should also identify certain properties for them. The following properties exist:
    - `SINGULAR()`: The predicate should only be true for one set of parameters at once across the entire problem. For example, `(robot_at ?l - location)` would be `SINGULAR()` since the robot can only be one location at a time. However, `(built ?h - house)` would not be `SINGULAR()` even though each house can only be built once since several different houses can be built at the same time.
    - `MAXONCE(?PAR)`: The predicate should be true at most once for the parameter ?PAR. For example, `(ball_in ?b - ball ?c - cup)` would be `MAXONCE(?b)` since each ball can only be in at most one cup. Only applicable if there are two or more parameters.
    - `SYMMETRICAL(?PAR1, ?PAR2)`: The predicate should always be used symmetrically. For example, `(connected ?i1 - intersection ?i2 - intersection)` would be `SYMMETRICAL(?i1, ?i2)` since if ?i1 is connected to ?i2 then ?i2 must also be connected to ?i1. As such, these should always be used in both directions, `(or (connected ?i1 ?i2) (connected ?i2 ?i1))`, in the domain. Only applicable if there are two or more parameters.
A predicate can have several properties at once. List the properties at the end of the predicate description.

Before defining the preconditions for an action, consider the implications of the action within the given domain. Identify any additional preconditions that are critical for the action to be performed successfully. Ensure that all necessary conditions are accounted for before listing them.

To support you, you're provided with the BNF of the supported PDDL grammar.

You must always go through the same steps and in the same level of detail as shown in the examples. Being long winded, overly detailed, or verbose is not an issue and instead encouraged.

As it's very important to remember, we'll reiterate this: the domain description always takes precedence over the action description. The action description might be wrong, or might have forgotten something but the domain description is always right.

# PDDL BNF

```
<predicate>                     ::= <name>
<variable>                      ::= ?<name>
<structure-def>                 ::= <action-def>
<typed list (x)>                ::= x*
<typed list (x)>                ::= :typing x+ - <type> <typed list(x)>
<primitive-type>                ::= <name>
<primitive-type>                ::= object
<type>                          ::= (either <primitive-type>+)
<type>                          ::= <primitive-type>
<emptyOr (x)>                   ::= ()
<emptyOr (x)>                   ::= x

<action-def>                    ::= (:action <action-symbol>
                                        :parameters (<typed list (variable)>)
                                        <action-def body>)
<action-symbol>                 ::= <name>
<action-def body>               ::= [:precondition <emptyOr (pre-GD)>]
                                    [:effect <emptyOr (effect)>]

<pre-GD>                        ::= <pref-GD>
<pre-GD>                        ::= (and\n\t<pre-GD>*)
<pre-GD>                        ::= (forall\n\t(<typed list(variable)>)\n\t<pre-GD>\n)
<pref-GD>                       ::= <GD>
<pref-name>                     ::= <name>
<GD>                            ::= <atomic formula(term)>
<GD>                            ::= <literal(term)>
<GD>                            ::= (and\n\t<GD>*\n)
<GD>                            ::= (or\n\t<GD>*\n)
<GD>                            ::= (not\n\t<GD>\n)
<GD>                            ::= (imply\n\t<GD>\n\t<GD>\n)
<GD>                            ::= (exists\n\t(<typed list(variable)>)\n\t<GD>\n)
<GD>                            ::= (forall\n\t(<typed list(variable)>)\n\t<GD>\n)

<literal(t)>                    ::= <atomic formula(t)>
<literal(t)>                    ::= (not\n\t<atomic formula(t)>\n)
<atomic formula(t)>             ::= (<predicate> t*)
<atomic formula(t)>             ::= (= t t)
<term>                          ::= <name>
<term>                          ::= <variable>

<name>                          ::= <letter> <any char>*
<letter>                        ::= a..z | A..Z
<any char>                      ::= <letter> | <digit> | - | _
<number>                        ::= <digit>+ [<decimal>]
<digit>                         ::= 0..9
<decimal>                       ::= .<digit>+

<effect>                        ::= (and\n\t<c-effect>*\n)
<effect>                        ::= <c-effect>
<c-effect>                      ::= (forall\n\t(<typed list (variable)>)\n\t<effect>\n)
<c-effect>                      ::= (when\n\t<GD>\n\t<cond-effect>\n)
<c-effect>                      ::= <p-effect>
<p-effect>                      ::= (not\n\t<atomic formula(term)>\n\t)
<p-effect>                      ::= <atomic formula(term)>
<cond-effect>                   ::= (and\n\t<p-effect>*\n)
<cond-effect>                   ::= <p-effect>
```
Note that all nested structures such as `or`, `and`, `not`, `exists`, `forall`, `imply`, and `when` should be written across several lines.

Note that you're unable to use constants. Instead, you should use predicates to implicitly model whatever you would attempt to use constants for. Additionally, you can only use boolean predicates and as such each predicate should be either true or false.

You are unable to create new types. You must only use the existing types.

## Example 1
I've got a building with five floors and two elevators, and I want you to plan the elevator trips for Bob at floor 3 and Sally at floor 2 to both go to floor 5 in a climate-friendly way. People can only enter, exit, or use elevators when the power of the building is on.

### Types
- object: Everything is an object
    - floor: Different levels in the building, numbered 1 to 5.
    - elevator: Elevators that can transport people between floors.
    - person: Individuals like Bob and Sally who need to travel to different floors.

floor is a/an object
elevator is a/an object
person is a/an object

### Available Predicates
- (on ?e - elevator ?f - floor): true if the elevator ?e is on the floor ?f. Properties: MAXONCE(?e)

### Action
drop_off

An elevator drops off a person. Requires that the person is inside the elevator. The person is then no longer inside the elevator but instead outside of it.

#### Action Parameters
The action states that refers to a *person* and an *elevator*. Let's define these:
```
- ?p - person: The person being dropped off
- ?e - elevator: The elevator dropping off
```

However, we also need to know which floor the *person* is exiting too. While this parameter isn't explicitly mentioned, we'll need it later so we have to define it now:
```
- ?f - floor: The floor where the person is being dropped off
```

#### Action Preconditions
The action description states that:
    1. The *person* must be inside the *elevator*. For this we need to define a new predicate, let's use `inside` and check `(inside ?p ?e)`.

Next, the domain description states something which seems to have been forgotten in the action description:
    2. To use an *elevator*, the power in the building must be on. As we're only modelling a single building, the power being on is a global property and we can use a single predicate without any parameters to model this. Let's use a `(power_on)` predicate for this.

Additionally, common sense states that we can only exit an *elevator* on the *floor* where the elevator currently is. So, we must check that the elevator is actually on the right floor:
    3. The *elevator* must be on the *floor* we're dropping off to. For this we can use the `on` predicate: `(on ?e ?f)`.

Beyond these, there are no explicitly stated, implicit or common sense preconditions on the action.

Let's summarize before writing code:
    - `?p` is inside `?e`: `(inside ?p ?e)`
    - The power must be on: `(power_on)`
    - `?e` must be at `?f`: `(on ?e ?f)`

Now, it's time to write the code:
```
(and ; All these must hold
    (inside ?p ?e) ; The person is inside the elevator
    (power_on) ; The power is on in the building
    (on ?e ?f) ; The elevator is on the floor
)
```

#### Action Effects
The action description says that:
    1. The *person* is no longer inside the *elevator*. For this we can use the `inside` predicate, setting `(not (inside ?p ?e))`.
    2. The *person* is now outside the *elevator*. This means that the *person* now instead is on the *floor*. The `on` predicate is currently only defined for *elevator*:s which are on *floor*:s. However, we can re-define it to allow for the first argument to be any *object*, though the intent will be for *elevator* or *person* to be used. All in all, this will give `(on ?p ?f)`.

Beyond these, there aren't any other effects mentioned by the domain description or which are commonsense. So, let's summarize what will happen:
    - `?p` is no longer in `?e`: `(not (inside ?p ?e))`
    - `?p` is now on `?f`: `(on ?p ?f)`

The code becomes:
```
(and ; All of this happens
    (not (inside ?p ?e)) ; The person is no longer in the elevator
    (on ?p ?f) ; The person is now on the floor
)
```

#### New Predicates
We used the following predicates:
    - `inside`
    - `power_on`
    - `on`

Let's define `inside` first. This models that a *person* is inside an *elevator*. Each *person* can only be inside one *elevator*, but each *elevator* can have any number of *person*:s inside it.
```
- (inside ?p - person ?e - elevator): true if the person ?p is inside the elevator ?e. Properties: MAXONCE(?p)
```

Next, the `power_on` predicate is a global predicate without any arguments which tracks if the power is on in the building.
```
- (power_on): true if the power is on in the building
```

Lastly, we also want to redefine the `on` predicate to allow it to take either an *elevator* or a *person*. Since their first joint type is *object*, we'll define it for that. It is still the case that it can be true at most once for each *elevator* or *person*, since they can't be at multiple places at once.
```
- (on ?o - object ?f - floor): true if the object ?o (a person or an elevator) is on floor ?f. Properties: MAXONCE(?o)
```

### Domain and Problem
In this domain, the AI agent is a wildlife conservation management system. It monitors animal populations within a reserve, tracks their movements using GPS collars, guides rangers, and generates relocation strategies for animals in overpopulated areas to ensure ecological balance. Movement can occur only between neighboring locations, and animals can only be in habitat locations. Furthermore, rangers can escort animals between habitats. Rangers can also guard locations where they are.

There are four habitats currently, one centered and three around it. The surrounding ones contain our deer and rabbits. However, there have been poachers seen recently, so I'd like to centralize all the deer and make sure that all the outer areas are protected by some of our rangers. All rangers are currently in the parking lot, which is connected to the center habitat.

### Types
- object: Everything is an object
    - animal: Animals within the reserve.
        - deer: A specific type of animal we wish to safeguard.
        - rabbit: Another type of animal in the reserve.
    - ranger: Workers who can perform various tasks.
    - location: Places within the reserve.
        - habitat: Specific areas where animals can live.

animal is a/an object
deer is a/an animal
rabbit is a/an animal

habitat is a/an location

### Available Predicates
- (at ?o - object ?l - location): true if the object ?o is at the location ?l. Properties: MAXONCE(?o)
- (guarding ?r - ranger ?l - location): true if the ranger ?r is guarding the location ?l. Properties: MAXONCE(?r)

### Action
move_animal

A ranger moves an animal from one location to another neighboring location. Requires the ranger to be in the location where the animal currently is and the target location to be neighboring. The animal and the ranger are then in the new location.

#### Action Parameters
First we'll need the *animal* to move and the *ranger* who moves it. Let's define these.
```
- ?r - ranger: The ranger moving the animal
- ?a - animal: The animal being moved
```

The action description also refers to a starting *location* and a target *location*. However, the domain description clearly states that animals can only be in *habitat*:s. The domain description is more important than the action description, and it seems like this was forgotten when writing the action description. Since we're moving animals, this should then clearly only be possible between *habitat*:s and not any *location*. Let's define these *habitat*:s.
```
- ?from - habitat: The habitat they're moving from
- ?to - habitat: The habitat they're moving to
```

I don't need any other implicit parameters.

#### Action Preconditions
To be able to move the *animal* to another *habitat* it's stated in the action description that:
    1: The *ranger* and *animal* are must both be in the starting *habitat*. We can address this using the already defined `at` predicate, requiring `(at ?r ?from)` and `(at ?a ?from)`.
    2: The two *habitat*:s must be neighboring. None of the existing predicates model this, so we'll have to create a new `neighboring` predicate and checking it. Since this is a bidirectional/symmetrical property, we'll check `(or (neighboring ?from ?to) (neighboring ?to ?from))` to be less sensitive to the initialization.

Additionally, the domain description states that "Rangers can also guard locations where they are". As such, if the *ranger* is currently guarding the *habitat*, it shouldn't be able to leave because if so it will be guarding a location where it isn't any longer. So, we get a third requirement:
    3: The *ranger* cannot be guarding the *habitat* `?from`. We can check this using the existing `guarding` predicate with `(not (guarding ?r ?from))`.

To summarize, the following is what we'll require:
    - `?r` is at `?from`: `(at ?r ?from)`
    - `?a` is at `?from`: `(at ?a ?from)`
    - `?from` is neighboring with `?to`: `(or (neighboring ?from ?to) (neighboring ?to ?from))`
    - `?r` is not guarding `?from`: `(not (guarding ?r ?from))`

Let's write the code for this:
```
(and ; All these conditions have to hold
    (at ?r ?from) ; The ranger must be at the location
    (at ?a ?from) ; The animal must be at the location
    (or ; The locations must be neighboring. For clarity we write this across multiple lines.
        (neighboring ?from ?to)
        (neighboring ?to ?from)
    )
    (not (guarding ?r ?from)) ; The ranger is not guarding the location it's leaving
)
```

#### Action Effects
When the *animal* is moved to a new *habitat* by the *ranger*, it's stated that:
    1. The *ranger* is now at the new *habitat*. For this we can use the existing `at` predicate with `(at ?r ?to)`
    2. The *animal* is now at the new *habitat*. Here we can use the existing `at` predicate with `(at ?a ?to)`

Additionally, using common sense and our knowledge of the domain we can understand that neither the *ranger* nor the *animal* can ever be at multiple places at once. As such, they should be removed from the old *habitat*
    3. The *ranger* is no longer at the old *habitat*. Here we can use the existing `at` predicate with `(not (at ?r ?from))`
    3. The *animal* is no longer at the old *habitat*. For this we can use the existing `at` predicate with `(not (at ?a ?from))`

Beyond these, there are no other effects. So, let's summarize:
    - `?r` is now at `?to`: `(at ?r ?to)`
    - `?a` is now at `?to`: `(at ?a ?to)`
    - `?r` is no longer at `?from`: `(not (at ?r ?from))`
    - `?a` is no longer at `?from`: `(not (at ?a ?from))`

Let's write the code for this:
```
(and ; All these effects happen
    (at ?r ?to) ; The ranger is now at the destination
    (at ?a ?to) ; The animal is now at the destination
    (not (at ?r ?from)) ; The ranger is no longer at the origin
    (not (at ?a ?from)) ; The animal is no longer at the origin
)
```

#### New Predicates
Now it's time to define the new predicates used in the **move_animal** action. We used:
    - `at`
    - `guarded`
    - `neighboring`

We don't need to redefine `at` or `guarded` since they already existed, though we could do so if we want to change their descriptions. However, the existing descriptions are fine.

We do need to define the `neighboring` predicate which models when two *habitats*:s are neighboring. Since we'll probably want to use this for *location* as well later, we'll define it for *location* rather than *habitat* however.. Since this is for a symmetrical/bidirectional property, we'll define it as `SYMMETRICAL` in the parameters. Let's define it:
```
- (neighboring ?l1 - location ?l2 - location): true if ?l1 and ?l2 are neighboring allowing movement between them. Properties: SYMMETRICAL(?l1, ?l2)
```

### Action
build_roof

A worker builds the roof at a house. Requires that the worker is at the location, that the worker isn't consumed, that all workers at the location are already consumed, that the roof is not already built. The effect is that the roof is now built.

#### Action Parameters
The action description refers to a *house* and a *worker* as well as a roof. However, roof isn't a type and rather a property of the *house* so we'll model that with a predicate later instead. So, let's define what we need:
```
- ?w - worker: The worker who should build the roof
- ?h - house: The house where the roof is to be built
```

Beyond these explicitly mentioned parameters, we shouldn't need any more.

#### Action Preconditions
The action description states the following requirements:
    1. The *worker* must be at the *location*. Clearly the "location" meant is the house. For this we can use the `at` predicate with `(at ?w ?h)`.
    2. That the *worker* isn't consumed. For this, we need to create a new predicate which tracks worker consumption, such as `consumed`. Then we can check `(not (consumed ?w))`.
    3. That all *worker*:s at the *location* are already consumed. However, this seems to be a mistake. The attempt is clearly to follow the domain description which says "when building anything all other workers at the location must be consumed already", but that is referring only to other *worker*:s! If we were to follow the action description verbatim, the action would never be applicable since the *worker* performing must both be at the location and not be consumed. Instead, we'll check that all other *worker*:s at the location are consumed. For this, we can use an `forall` operator with an `imply` to that each other worker at the *house* is consumed.
    4. That the roof is not already built. For this we'll need to add a new `roof_built` predicate and then check `(not (roof_built ?h))`.

However, the domain description states that both walls and roofs exists and it doesn't make sense if the roof is built before the walls. We need to address this too:
    5. That the walls are already built. For this we'll need to add a new `walls_built` predicate and require `(walls_built ?h)`.

Let's summarize before we write code. We need to check:
    - `?w` is at `?h`: `(at ?w ?h)`
    - `?w` isn't consumed: `(not (consumed ?w))`
    - No un-consumed *worker*:s are at the location: An `forall` condition over the *worker* type combined with an `imply`
    - The roof is not already built: `(not (roof_built ?h))`
    - The walls are already built: `(walls_built ?h)`

The code for this becomes:
```
(and ; All the following must hold

            (and ; Both these conditions hold
                (not (= ?w_other ?w)) ; ?w_other is actually another worker. This must be explicitly checked, just the name isn't enough
                (at ?w_other ?h) ; The other worker is also at the house
            )
            (consumed ?w_other) ; Then the other worker must be consumed
        )
    )
    (not (roof_built ?h)) ; The roof is not already built
    (walls_built ?h) ; The walls are already built
)
```

#### Action Effects
When the action is applied, the action description states that:
    1. The roof is now built. For this we can use the `roof_built` predicate from above. We'll set `(roof_built ?h)`.

Additionally however, the domain description states that "Building anything consumes the worker". This must have been forgotten in the action description, but should still occur:
    2. The worker is now consumed. Here we can use `consumed`, using `(consumed ?w)`.

Beyond these, no other effects are stated or make sense. So, let's write summarize:
    - The roof of `?h` is now built: `(roof_built ?h)`
    - `?w` is now consumed: `(consumed ?w)`

Now, let's write the code:
```
(and ; All these effects happen

```
