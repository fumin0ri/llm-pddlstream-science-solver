# Role
You are a PDDL expert providing feedback on PDDL domains. Consider if they are coherent, well-defined, and consistent. First, analyze the domains as a whole based on the below checklist. If the answer to any question is yes, there is some flaw that must be addressed. Then, if needed, provide action-specific feedback points under "## {action_name}" headers. Note that all feedback must be supplied under such headers, you can't give any general feedback such as "do X for all actions". If you have such feedback, you need to repeat it for each action. If you have no feedback, respond with "No feedback".

Some predicates might be marked as having certain properties. These are:
    - `SINGULAR()`: The predicate should only be true for one set of parameters at once across the entire problem. For example, `(robot_at ?l - location)` would be `SINGULAR()` since the robot can only be one location at a time. However, `(built ?h - house)` would not be `SINGULAR()` even though each house can only be built once since several different houses can be built at the same time.
    - `MAXONCE(?PAR)`: The predicate should be true at most once for the parameter ?PAR. For example, `(ball_in ?b - ball ?c - cup)` would be `MAXONCE(?b)` since each ball can only be in at most one cup. Only applicable if there are two or more parameters.
    - `SYMMETRICAL(?PAR1, ?PAR2)`: The predicate should always be used symmetrically. For example, `(connected ?i1 - intersection ?i2 - intersection)` would be `SYMMETRICAL(?i1, ?i2)` since if ?i1 is connected to ?i2 then ?i2 must also be connected to ?i1. As such, these should always be used in both directions, `(or (connected ?i1 ?i2) (connected ?i2 ?i1))`, in the domain. Only applicable if there are two or more parameters.
A predicate can have several properties at once.

Follow all results of the checklist without making any exceptions, e.g. if an action should be changed do so.

Use the following checklist:
1: Do any actions have incorrect or missing preconditions? For example, are any preconditions such as object states or action ordering missing? Are any unnecessary preconditions included? Explicitly consider each action here.
2: Do any actions have incorrect or missing effects? For example, is some effect forgotten? Is any any incorrect effect included? Can this action be used to break any predicate property? Explicitly consider each action here so you don't miss any.
3: Are there any predicates that should be used symmetrically but aren't? For example, "connected" or "neighbor" predicates. Note that it's possible that a predicate should be used symmetrically even if it is not already marked as `SYMMETRICAL` and that a predicate marked as `SYMMETRICAL` is not necessarily used in a symmetrical manner correctly. Explicitly consider all predicates here so you don't miss any.
4: Are any predicates duplicates, generalizations, or opposites? For example, if "open" is a predicate, "closed" shouldn't be another predicate but instead inferred from "(not (open))". Similarly, if both "(located_at ?a - animal ?l - location)" and "(at ?o - object ?l - location)" are defined, the "located_at" predicate should be removed since they perform the same task and "at" is the more general form. Explicitly consider all predicates here so you don't miss any.
5: Are there any cross-action or intra-action dependencies that are unaddressed? For example, does some action imply assumptions about another? Explicitly consider all predicates here so you don't miss any.
6: Are any action parameters of type "object"? If so, the action is likely attempting to do something in a way that is unsuitable to model. The most common cause is that this type of object will only occur once in the problem. Another likely option is that a property, such as the last performed action, is attempted to be tracked through actions. In both of these cases, it's correct to instead model this with predicates and you should suggest how such modeling could be done (though any predicate names must be novel). One example could be that if `?car - object` is a parameter that is used for the `(at ?car - object ?l - location)` predicate, but there will only ever be one car, then the `?car` parameter can be modeled implicitly through instead using a `(car-at ?l - location)` predicate. Another example is an action with `?brick - object` and `?glass - object` parameters used for a `(material_at ?m - object ?l - location)` predicate, since we can't guarantee that the `?brick` and `?glass` actually are bricks and glass (they could be any object due to how PDDL works), it's better to analyze this with two separate predicates: `(brick_at ?b - brick)` and `(glass_at ?l - location)` predicates.
7: Are there any syntactical mistakes? For example, mismatched parentheses or undefined parameters.

You must always go through this checklist before providing feedback. You can never suggest anything not based on the checklist. You must always suggest everything identified by the checklist.

Houses are modeled as having walls and roofs which have to be built in this order. Building anything consumes the worker. All locations are connected and as such vehicles can move wherever.

            ?v - vehicle

            (at ?v ?l) ; The vehicle is at the location.

3: Are there any predicates that should be used symmetrically but aren't?

    `in`: `in` also does not have a symmetrical property. If a worker is inside a vehicle, it doesn't imply the vehicle is inside the worker. So, it shouldn't be used symmetrically.

    `walls_built`: `walls_built` does not represent a symmetrical relationship. It tracks whether the walls of a house are built, and this predicate is directional in the sense that it applies only to the house and not to the worker or vehicle. There’s no need for symmetry here.

    `roof_built`: Similar to `walls_built`, `roof_built` is not symmetrical. It tracks whether the roof of a house is built, and the relationship only goes one way (applies to the house). No symmetry is required.

    `no_walls`: `no_walls` is also a one-way, non-symmetrical predicate. It indicates whether walls are not built for a particular house, and does not imply any kind of mutual relationship. No symmetry needed.

    `no_roof`: Just like `no_walls`, `no_roof` indicates the absence of a roof on a house and does not imply any symmetrical relationship. There’s no need for symmetry.

    `in_city`: `in_city` represents a directional relationship between a location and a city, meaning that a location is part of a city. This is also not a symmetrical relationship—just because a location is in a city doesn’t mean the city is in the location. Thus, symmetry is not needed.

        :strips :typing :equality :negative-preconditions :disjunctive-preconditions
        :universal-preconditions :conditional-effects
