# Role
You are a PDDL expert, and your task is to define the initial state and the goal state for a PDDL problem based on a domain description and the available predicates. For the initial state specify both object instances and which predicates are true, false predicates don't have to be specified. For the goal, specify the states that need to have specific values regardless if those are true or false. Do it step-by-step and explain your thoughts. Respond with the exact headings provided. You can't assume that any object, regardless of type, already exists. Everything you wish to use should be defined here. Also, remember that any symmetrical predicates should be defined both ways.

Reply using exactly the following headers:
- "## Object Instances": The objects which exist in the domain. Each object should be within a markdown block and on its own line. The type and description should be specified.
- "## State": List which predicates are initially true. False predicates do not need to be stated. Each predicate should be within a markdown block on its own line.
- "## Goal": Include a single markdown code block that details the goal expression in PDDL code.

Some predicates might be marked with properties. If so, these should be followed during initialization. The possible properties are:
    - `SINGULAR()`: The predicate should only be true for one set of parameters at once across the entire problem. For example, `(robot_at ?l - location)` would be `SINGULAR()` since the robot can only be one location at a time. However, `(built ?h - house)` would not be `SINGULAR()` even though each house can only be built once since several different houses can be built at the same time.
    - `MAXONCE(?PAR)`: The predicate should be true at most once for the parameter ?PAR. For example, `(ball_in ?b - ball ?c - cup)` would be `MAXONCE(?b)` since each ball can only be in at most one cup. Only applicable if there are two or more parameters.
    - `SYMMETRICAL(?PAR1, ?PAR2)`: The predicate should always be used symmetrically. For example, `(connected ?i1 - intersection ?i2 - intersection)` would be `SYMMETRICAL(?i1, ?i2)` since if ?i1 is connected to ?i2 then ?i2 must also be connected to ?i1. As such, these should always be defined in both directions, `(connected ?i1 ?i2)\n(connected ?i2 ?i1)`. Only applicable if there are two or more parameters.
A predicate can have several properties at once. However, a predicate might have a property, such as being symmetrical, without being marked as such.

You must always go through the same steps and in the same level of detail as shown in the examples. Being long winded, overly detailed, or verbose is not an issue and instead encouraged.

# Examples
## Example 1
### Domain and Problem
I've got a building with five floors and two elevators, and I want you to plan the elevator trips for Bob at floor 3 and Sally at floor 2 to both go to floor 5 in a climate-friendly way. People can only enter, exit, or use elevators when the power of the building is on. The power is initially on.

### Types
- object: Everything is an object
    - floor: Different levels in the building, numbered 1 to 5.
    - elevator: Elevators that can transport people between floors.
    - person: Individuals like Bob and Sally who need to travel to different floors.

floor is a/an object
elevator is a/an object
person is a/an object

### Predicates
- (inside ?p - person ?e - elevator): true if the person ?p is inside the elevator ?e. Properties: MAXONCE(?p)
- (power_on): true if the power is on in the building
- (on ?o - object ?f - floor): true if the object ?o (a person or an elevator) is on floor ?f. Properties: MAXONCE(?o)

### Object Instances
The problem states that there are five *floor*:s in the building. Let's define these:
```
floor1 - floor ; The first floor
floor2 - floor ; The second floor
floor3 - floor ; The third floor
floor4 - floor ; The fourth floor
floor5 - floor ; The fifth floor
```

Next, there are two *elevator*:s.
```
elevator1 - elevator ; The first elevator
elevator2 - elevator ; The second elevator
```

Lastly, the description says that there are two people, modelled with *person*, Bob and Sally.
```
bob - person ; One of the people traveling
sally - person ; The other of the people traveling
```

We've now used all the types and defined all explicitly mentioned objects. There is no reason to believe that any more objects are needed, so let's move on to the initial state.

### State
For the `inside` predicate, we define all people who start inside an *elevator*. No one starts inside any *elevator*, so we don't initialize any of these.

The `power_on` predicate should start true if the power is on initially, which the domain description states it is. So, let's state this.
```
(power_on) ; The power starts on
```

Next, the `on` predicate defines on which *floor* the *elevator*:s and *people* start. We don't know where the *elevator*:s start, but we can guess that they have to start somewhere for the domain to work. Since we don't know and it's common for *elevator*:s to wait on the bottom *floor*, let's assume that they start at `floor1`. For `sally` and `bob` we know where they start.
```
(on elevator1 floor1) ; We assume that elevator1 starts at floor1
(on elevator2 floor1) ; We assume that elevator2 starts at floor1
(on bob floor3) ; Bob starts at floor 3
(on sally floor3) ; Sally starts at floor 2
```

We've now inspected all the predicates, and it doesn't seem like there is anything missing. There are no additional predicates to initialize, so it's time for the goal.

### Goal
The goal states that we want `sally` and `bob` to meet at `floor5`. We do not care about anything else, such as the power state or where the *elevator*:s are. This goal becomes:
```
(and ; All this must hold
    (at bob floor5) ; Bob is on floor5
    (at sally floor5) ; Sally is on floor5
)
```

## Example 2
### Domain and Problem
In this domain, the AI agent is a wildlife conservation management system. It monitors animal populations within a reserve, tracks their movements using GPS collars, guides rangers, and generates relocation strategies for animals in overpopulated areas to ensure ecological balance. Movement can occur only between neighboring locations, and animals can only be in habit locations. Rangers can also guard locations where they are.

There are four habitats currently, one centered and three around it. The surrounding ones contain our deer. However, there have been poachers seen recently so I'd like to centralize all the deer and make sure that all the outer areas are protected by some of our rangers all of which start in the center.

### Types
- object: Everything is an object
    - movable: Animals and rangers which can move through the conservatory
        - animal: Animals within the reserve.
            - deer: A specific type of animal we wish to safeguard.
            - rabbit: Another type of animal in the reserve.
        - ranger: Workers who can perform various tasks.
    - location: Places within the reserve.
        - habitat: Specific areas where animals can live.

movable is a/an object
animal is a/an movable
deer is a/an animal
rabbit is a/an animal
ranger is a/an movable
location is a/an object
habitat is a/an location

### Predicates
- (neighboring ?l1 - location ?l2 - location): true if ?l1 is neighboring ?l2. Properties: SYMMETRICAL(?l1, ?l2)
- (at ?m - movable ?l - location): true if the movable ?m (an animal or ranger) is at the location ?l. Properties: MAXONCE(?m)
- (guarding ?r - ranger ?l - location): true if the ranger ?r is guarding the location ?l. Properties: MAXONCE(?r)

### Object Instances
The text says that there are four *habitat*:s, one in the center and three around it. Let's define all of these.
```
center - habitat ; The center habitat
outer1 - habitat ; The first outer habitat
outer2 - habitat ; The second outer habitat
outer3 - habitat ; The third outer habitat
```

Additionally, it's described that there are *deer* in all the surrounding *habitat*:s. Since there are three surrounding *habitat*:s, we'll need three *deer*.
```
deer1 - deer ; The deer starting in outer1
deer2 - deer ; The deer starting in outer2
deer3 - deer ; The deer starting in outer3
```

The text also mentions "poachers", but these aren't in the type hierarchy and not modelled with any predicate, so they shouldn't be modelled by us.

Next, the text mentions that there are several *ranger*:s. The exact number of *ranger*:s isn't specified exactly, but since all outer locations have to be guarded and `guarding` is `MAXONCE(?r)` each *ranger* can only guard a single location so we'll need at least three to perform the guarding. However, this is only stated to be "some of our rangers", so there must be more. Let's define twice that many as an estimate.

```
ranger1 - ranger ; The first ranger
ranger2 - ranger ; The second ranger
ranger3 - ranger ; The third ranger
ranger4 - ranger ; The fourth ranger
ranger5 - ranger ; The fifth ranger
ranger6 - ranger ; The sixth ranger
```

There are several types we haven't used explicitly (*object*, *movable*, *animal*, *deer*, and *location*). However, none of these are mentioned in the description and there is no common sense reason why we would need any of these. So, we don't include any.

### State
Let's go through all the objects and predicates to see what should be initialized.

For all the *habitat*:s, it's stated that the `center` habitat is in the middle of the others. We can assume that this means that it is `neighboring` with all `outer` locations, but that none of the `other` locations are adjacent to each other. Since `neighboring` is symmetrical, we should define this in both directions for each pair. Let's define it:
```
(neighboring center outer1) ; center is adjacent to outer1
(neighboring outer1 center) ; outer1 is adjacent to center
(neighboring center outer2) ; center is adjacent to outer2
(neighboring outer2 center) ; outer2 is adjacent to center
(neighboring center outer3) ; center is adjacent to outer3
(neighboring outer3 center) ; outer3 is adjacent to center
```

Next, we need to specify where all the *deer* start. There is one *deer* starting at each `outer` *habitat*, so let's define this.
```
(at deer1 outer1) ; deer1 starts at outer1
(at deer2 outer2) ; deer2 starts at outer2
(at deer3 outer3) ; deer3 starts at outer3
```

Then, the initial location of each *ranger* must be specified. The description states that they all start at the `center`. This becomes:
```
at(ranger1, center) ; ranger1 starts at center
at(ranger2, center) ; ranger2 starts at center
at(ranger3, center) ; ranger3 starts at center
at(ranger4, center) ; ranger4 starts at center
at(ranger5, center) ; ranger5 starts at center
at(ranger6, center) ; ranger6 starts at center
```

We've now initialized the `neighboring` predicate for all pairs of neighboring *location*:s as well as the initial position of each *deer* and *ranger*. The only predicate we've not fully initialized is `guarding`, but there is no reason to believe that any ranger starts guarding any location. Instead, guarding the `outer` location` is the goal. So, let's move on to defining the goal.

### Goal
Part of the goal is that each `outer` *location* should be guarded. However, it's not important which *ranger* guards the respective locations, only that someone does so. We can check for this with the `exists` keyword. The other half of the goal is that all the `deer` should be moved to the `center` location, for this we can use `forall`. The goal then becomes:
```
(and ; All the following must hold
    (exists (?ranger_guarding_outer1 - ranger) ; There is some ranger
        (guarding ?ranger_guarding_outer1 outer1) ; Who is guarding outer1
    )
    (exists (?ranger_guarding_outer2 - ranger) ; There is some ranger
        (guarding ?ranger_guarding_outer2 outer2) ; Who is guarding outer2
    )
    (exists (?ranger_guarding_outer3 - ranger) ; There is some ranger
        (guarding ?ranger_guarding_outer3 outer3) ; Who is guarding outer3
    )
    (forall (?d - deer) ; All deers
        (at ?d center) ; Should be in the center
    )
)
```

## Example 3
### Domain and Problem
The AI agent is here a scheduling system for a house construction company with administrators who can create orders, workers who perform them, and vehicles to transport workers.

Currently, there are a couple of trucks at the Chicago depot and we're trying to build three houses in the area. Jamie is the only administrator, but we've got Emma and Bob available for general work.

### Types
- object: Everything is an object
    - worker: Human workers who do things
        - administrator: Workers who can create orders for other workers
        - general_worker: Workers who can perform a variety of tasks, if there is an order for it
    - vehicle: Used to transport workers
    - location: Places where workers and machines can be
        - house: What is constructed.

movable is a/an object
worker is a/an movable
administrator is a/an worker
construction_worker is a/an worker
vehicle is a/an movable
location is a/an object
house is a/an location

### Predicates
- (at ?o - object ?l - location): true if the object ?o (which should be a worker or vehicle) is at location ?l. Properties: MAXONCE(?o)
- (inside ?w - worker ?v - vehicle): true if the worker ?w is in the vehicle ?v. Properties: MAXONCE(?o)
- (wall_order ?h - house): true if there is an order to build a wall at house ?h
- (wall_built ?h - house): true if a wall has been built for house ?h
- (floor_order ?h - house): true if there is an order to build a floor at house ?h
- (floor_built ?h - house): true if a floor has been built for house ?h
- (floor_order ?h - house): true if there is an order to build a roof at house ?h
- (roof_built ?h - house): true if a roof has been built for house ?h
- (finalized ?h - house): true if the house ?h has been completed.
- (connected ?l1 - location ?l2 - location): true if there is a road between l1 and l2 allowing travel between them.

### Object Instances
We assume that there are two trucks since the description said "a couple".
```
truck1 - vehicle; the first truck at the Chicago depot
truck2 - vehicle; the second truck at the Chicago depot
```

Next, let's define the relevant locations. Since "chicago_depot" is not a house, we model it as a general location.
```
chicago_depot - location; The Chicago depot
house1 - house; The first house to build
house2 - house; The second house to build
house3 - house; The third house to build
```

Now, let's model the worker.
```
jamie - administrator; The administrator Jamie
emma - general_worker; The first worker, Emma
bob - general_worker; The second worker, Bob
```

### State
Let's start by specifying where everyone is. It's a reasonable assumption that everyone starts at the Chicago depot.
```
(at truck1 chicago_depot); truck1 is at the chicago_depot
(at truck2 chicago_depot); truck2 is at the chicago_depot
(at jamie chicago_depot); Jamie is at the depot
(at emma chicago_depot); Emma is at the depot
(at bob chicago_depot); Bob is at the depot
```

We assume that all the houses are entirely unbuilt, that the trucks are unloaded and that no orders are written. As such, all these conditions are false and don't need to be specified.

Finally, we need to specify how the various locations are connected. Let's assume that all the connections are through the depot acting as a hub.
```
(connected house1 chicago_depot); house1 is connected to the chicago_depot
(connected house2 chicago_depot); house2 is connected to the chicago_depot
(connected house3 chicago_depot); house3 is connected to the chicago_depot
```

Finally, the "connected" predicate states that two locations are connected if there exists a road between them. So, clearly if (connected A B) then (connected B A) since roads go both ways, even if "connected" wasn't marked with the `SYMMETRICAL` property. So, we need to mirror all the "connected" constraints.
```
(connected chicago_depot house1); chicago_depot is connected to house1
(connected chicago_depot house2); chicago_depot is connected to house2
(connected chicago_depot house3); chicago_depot is connected to house3
```

### Goal
The stated goal is that all houses should be finalized. As such, this is what we need to check.

```
(and ; all the following should be done
   (finalized house1) ; house 1 is done
   (finalized house2) ; house 2 is done
   (finalized house3) ; house 3 is done
)
```
