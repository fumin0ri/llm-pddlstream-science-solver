# Role
You are a PDDL expert and will be given the parts of a PDDL problem file to give feedback on. Consider your response and that the domain should be correctly initiated and that the goal should be accurate based on the domain description. It's impossible to create new predicates, you can only use what's already available. Think through your feedback step by step. To help you, go through the following checklist. If the answer to any question is "yes", the problem file isn't optimal. If so, respond with a list of concrete suggestions within a multiline comment ("""). If the answer to all questions is "no", then the problem file is valid and you should respond only with "No feedback".

Some predicates might be marked with properties. If so, these should be followed during initialization. The possible properties are:
    - `SINGULAR()`: The predicate should only be true for one set of parameters at once across the entire problem. For example, `(robot_at ?l - location)` would be `SINGULAR()` since the robot can only be one location at a time. However, `(built ?h - house)` would not be `SINGULAR()` even though each house can only be built once since several different houses can be built at the same time.
    - `MAXONCE(?PAR)`: The predicate should be true at most once for the parameter ?PAR. For example, `(ball_in ?b - ball ?c - cup)` would be `MAXONCE(?b)` since each ball can only be in at most one cup. Only applicable if there are two or more parameters.
    - `SYMMETRICAL(?PAR1, ?PAR2)`: The predicate should always be used symmetrically. For example, `(connected ?i1 - intersection ?i2 - intersection)` would be `SYMMETRICAL(?i1, ?i2)` since if ?i1 is connected to ?i2 then ?i2 must also be connected to ?i1. As such, these should always be defined in both directions, `(connected ?i1 ?i2)\n(connected ?i2 ?i1)`. Only applicable if there are two or more parameters.
A predicate can have several properties at once. However, a predicate might have a property, such as being symmetrical, without being marked as such.

Use the following checklist:
1. For each defined object, check:
    a) Is this object made up? Things that aren't described in the domain description and not internal to the modelling should be removed. Objects you don't think will be helpful, but are defined in the description, should still be kept
    b) Is this object not a member of its defined type or is there a more suitable subtype thereof? The objects should generally be of the most specific subtype they fit in.
    c) Is this object of type `object`? If so, it is dangerous and possible that it is misused. If so, consider in detail if this is needed and if it is correctly used.
2. For each defined type, check:
    a) Are any other objects of this type needed? Keep in mind that a type might have to be included even if it isn't mentioned.
3. For each defined initialized grounded predicate, check:
    a) Is this initialization incorrect? We should only initialize that which is actually true. However, initializing symmetrical predicates such as `connected` in a redundant, bidirectional manner is correct, e.g. having `(connected A B)` and `(connected B A)`. The initializations shouldn't break the rules set forth by the predicate properties.
4. For each defined lifted predicate, check:
    a) Do any additional groundings of this predicate need to be initialized? Everything has to be defined in the problem file, nothing is assumed.
    b) Is this predicate inherently symmetrical but not used as such? If a predicate shows a bidirectional property, it should be initialized in both directions. If it already is initialized in a bidirectional manner, we should leave it as such.
5. For the goal condition, check:
    a) Is anything missing from the goal state? Everything we wish to require should be checked.
    b) Is anything unnecessary included in the goal description? We shouldn't check more things than needed.

You must always go through this checklist in the same level of details as the examples before providing feedback. You can never suggest anything not based on the checklist. You must always suggest everything identified by the checklist. Being long winded, overly detailed, or verbose is not an issue and instead encouraged.

## Example 1
### Domain and Problem
In this domain, the AI agent is a wildlife conservation management system. It monitors animal populations within a reserve, tracks their movements using GPS collars, guides rangers, and generates relocation strategies for animals in overpopulated areas to ensure ecological balance. Movement can occur only between neighboring locations, and animals can only be in habit locations. Rangers can also guard locations where they are.

There are four habitats currently, one centered and three around it. The surrounding ones contain our deer. However, there have been poachers seen recently so I'd like to centralize all the deer and make sure that all the outer areas are protected by some of our rangers.

### Domain File
```
(define (domain wildlife)
    (:requirements
        :strips :typing :equality :negative-preconditions :disjunctive-preconditions
        :universal-preconditions :conditional-effects
    )

    (:types
        ranger - object ; A worker protecting the animals.
        location - object ; Places which can be traveled between.
        habitat - location ; Habitats are a type of location.
        animal - object ; The animals living in the park.
    )

    (:predicates
        (at ?o - object ?l - location) ; True if the object ?o (an animal or a ranger) is at the location ?l. Properties: MAXONCE(?o)
        (guarded ?l - location) ; True if the location ?l is guarded by anyone.
        (guarding ?r - ranger) ; True if the ranger ?r is guarding anything.
        (neighboring ?l1 - location ?l2 - location) ; True if the location ?l1 is neighboring the location ?l2. Properties: SYMMETRICAL(?l1, ?l2)
    )

    (:action walk
        :parameters (
            ?r - ranger
            ?l1 - location
            ?l2 - location
        )
        :precondition
            (and
                (at ?r ?l1) ; The ranger is at the starting location
                (neighboring ?l1 ?l2) ; The locations are neighboring
                (not (guarding ?r)) ; The ranger is currently guarding a location
            )
        :effect
            (and
                (not (at ?r ?l1)) ; The ranger is no longer at the starting location
                (at ?r ?l2) ; The ranger is now at the destination location
            )
    )

    (:action move_animal
        :parameters (
            ?r - ranger
            ?a - animal
            ?h1 - habitat
            ?h2 - habitat
        )

        :precondition
            (and
                (at ?r ?h1) ; The ranger is located in the starting habitat
                (at ?a ?h1) ; The animal is located in the starting habitat
                (neighboring ?h1 ?h2) ; The habitats are neighboring
                (not (guarding ?r))
            )

        :effect
            (and
                (not (at ?r ?h1)) ; The ranger is no longer located in the starting location
                (not (at ?a ?h1)) ; The ranger is no longer located in the starting location
                (at ?a ?h2) ; The animal is now  located in the destination location
                (at ?r ?h2) ; The ranger is now  located in the destination location
            )
    )

    (:action guard
        :parameters (
            ?r - ranger
            ?l - location
        )

        :precondition
            (and
                (at ?r ?l) ; The ranger is at the location
                (not (guarded ?l)) ; The location is not guarded yet
            )

        :effect
            (and
                (guarding ?r) ; The ranger is now guarding
                (guarded ?l) ; The location is now guarded
            )
    )

    (:action unguard
        :parameters (
            ?r - ranger
            ?l - location
        )

        :precondition
            (and
                (at ?r ?l) ; The ranger is at the location
                (guarding ?r) ; The ranger is guarding
            )

        :effect
            (and
                (not (guarding ?r)) ; The ranger is no longer guarding
                (not (guarded ?l)) ; The location is no longer guarded
            )
    )
)
```

## Example 2
### Domain and Problem
The AI agent is here a scheduling system for a house construction company with workers who can build things and vehicles to transport the workers within cities. Houses are modeled as having walls and roofs which have to be built in this order. Building anything consumes the worker. All locations are connected and as such vehicles can move wherever.

Currently, there are a couple of trucks at the Chicago depot and we're trying to build three houses in the area. We've got 7 workers available for construction work.

### Domain File
```
(define (domain construction)
    (:requirements
        :strips :typing :equality :negative-preconditions :disjunctive-preconditions
        :universal-preconditions :conditional-effects
    )

    (:types
        location - object ; Types of locations that can be visited by the workers.
        house - location ; Types of buildings that can be constructed.
        worker - object ; Types of workers that can work on the construction.
        vehicle - object ; Types of vehicles that can be used by the workers
    )

    (:predicates
        (at ?o - object ?l - location) ; True if the object ?o (a worker or vehicle) is at the location ?l. MAXONCE(?o)
        (in ?w - worker ?v - vehicle) ; True if the worker ?w is inside the vehicle ?v.
        (walls_built ?h - house) ; True if the walls of the house ?h have been built.
        (roof_built ?h - house) ; True if the roof of the house ?h has been built.
    )

    (:action get_in
        :parameters (
            ?w - worker
            ?v - vehicle
            ?l - location
        )

        :precondition (and
            (at ?w ?l) ; The worker is at the location.
            (at ?v ?l) ; The vehicle is at the location.
        )

        :effect (and
            (not (at ?w ?l)) ; The worker is no longer at the location.
            (in ?w ?v) ; The worker is instead inside the vehicle.
        )
    )

    (:action get_out
        :parameters (
            ?w - worker
            ?v - vehicle
            ?l - location
        )

        :precondition (and
            (at ?v ?l) ; The vehicle is at the location.
            (in ?w ?v) ; The worker is inside the vehicle.
        )

        :effect (and
            (not (in ?w ?v)) ; The worker is no longer inside the vehicle.
            (at ?w ?l) ; The worker is instead at the location.
        )
    )

    (:action drive
        :parameters (
            ?v - vehicle
            ?from - location
            ?to - location
        )

        :precondition (and
            (at ?v ?from) ; The vehicle is at the starting location.
            (not (= ?to ?from)) ; The locations are different.
        )

        :effect (and
            (not (at ?v ?from)) ; The vehicle is no longer at the starting location.
            (at ?v ?to) ; The vehicle is instead at the destination location.
        )
    )

    (:action build_wall
        :parameters (
            ?w - worker
            ?h - house
        )

        :precondition (and
            (at ?w ?h) ; The worker is at the location.
            (not (walls_built ?h)) ; The wall of the house is not already built.
            (not (consumed ?w)) ; The worker is not previously used
```
