# Role
You are a PDDL expert and your task is to evaluate if a set of types is correct and sufficient for modeling a given domain. To help you, go through the following checklist. If the answer to any question is "yes", the types aren't sufficient. If so, respond with a list of concrete suggestions within a markdown code block. If the answer to all questions is "no", then the types are valid and you should respond only with "No feedback". Do not add any suggestions unrelated to the provided checklist. Follow all results of the checklist without making any exceptions, i.e. if a type violates one of the requirements it should be removed even if it might appear useful. Never suggest any new types that aren't supported by the checklist.

Go through the following checklist:
1: Are any types the exact same? If so, only one should be kept. A type being a metatype to another is okay, but they should not be synonyms or identical. Any types that have or enable different actions or behaviors should be kept separately. However, if the types have the same behavior only one type should be kept.
2: Are any of the types actually objects?  For example, is a name listed as a type? If so, it should be replaced with the general type.
3: Are any of the types actually actions? If so, they should be removed.
4: Are any of the types actually properties, relationships, connections, or otherwise better modeled using predicates (not counting `is_{type}` predicates)? If so, they should later be modeled with predicates instead and thereby removed.
5: Is the acting agent itself or the resulting plans included? If the acting agent is a robot or something physically embodied it can be kept however.
6: Is it likely that there will only be a single instance of a given non-subtype in the current problem (not counting initial/goal states)? If so, it should be removed. Explicitly consider each type. However, if a type will only be used once, but it is a subtype of another type that has more than one member, this type should still be kept. If there are several related types, it's also possible to suggest a metatype for these.Additionally, if the type is the physical embodiment of the acting agent, such as robot, it should be kept.
7: Are any of the types modeling states (this could be initial states, goal states, or other large parts of states)? The domain might be better modeled in PDDL if some parts are modeled not as objects and types, even if they technically are.
8: Are any of the types unrelated to the domain being modeled?

You must always go through this checklist before providing feedback. You can never suggest anything not based on the checklist. You must always suggest everything identified by the checklist.

# Examples

## Example 1
### Domain and Problem
In this domain, the AI agent is a wildlife conservation management system. It monitors animal populations within a reserve, tracks their movements using GPS collars, analyzes habitat conditions, and generates relocation strategies for animals in overpopulated areas to ensure ecological balance.

### Types
- location: Places which can be traveled between. Locations can contain other locations.
- habitat: A type of location with a distinct living style.
- starting_positions: Where the animals start.
- animal: An animal living within the reserve.
- target_positions: Where we wish the animals to end.
- rabbit: A small jumpy animal.
- relocation_strategy: How to transport the animals.
- conservatory: The conservation the AI is protecting.
- creature: The creatures living within the reserve.
- hunger: Measures how hungry an animal is.
- path: A connective road between two locations.

### Checklist
1: Are any types the exact same?
    The *animal* and *creature* types both model the inhabitants of the park. As such, only one should be kept. Since *animal* is the more common and grounded term, this should be kept.

    *habitat* is a type of *location*, however, the related actions will differ with *habitat* being the only locations animals can be in. Additionally, subtypes existing are valid and not a problem. So this is valid and should be kept.

    The *rabbit* type is a subtype of *animal*, and while some properties might differ there are no actions within this domain that will differ for a *rabbit* and an *animal*. As such, the *rabbit* type should be removed and only the *animal* type kept.

    Thereby: Yes.

2: Are any of the types actually objects?
    All the types are truly types. As such: No.

3: Are any of the types actually actions?
    None of the types are actions. Hence: No.

4: Are any of the types actually properties, relationships, connections, or otherwise better modeled using predicates (not counting `is_{type}` predicates)?
    The *hunger* type clearly measures a property. This ought to be controlled through predicates at a later stage, and as such not included now. Similarly, the *path* is more suitable to model with a `path_between` predicate or similar as the paths themselves are not interesting. Thereby: Yes.

5: Is the acting agent itself or the resulting plans included?
    The *relocation_strategies* are what the agent produces. As such, this shouldn't be included. Due to this: Yes.

6: Is it likely that there will only be a single instance of a given non-subtype in the current problem (not counting initial/goal states)?
    The *conservation* class refers to the exact conservation the AI is protecting, and as such there will only ever be one such instance. Thereby, there is no need to model it as a type. The remaining objects will likely be used several times, but as further details are unavailable this will not be discussed further. Hence: Yes.

7: Are any of the types modeling states (this could be initial states, goal states, or other large parts of states)?
    The *starting_positions* type actually describes the initial state. This should later be defined with for example an `(at obj - object loc - location)` predicate in the initial state instead.
    Similarly, the *target_positions* is better described in the goal state using the same `(at obj - object loc - location)` predicate. So: Yes.

8: Are any of the types unrelated to the domain being modeled?
    All types included are related to wildlife conservation. As such: No.

### Feedback
I will now provide suggestions based exclusively, completely, and only according to the checklist. I will not suggest anything not identified by the checklist.
```
- Remove the following types:
    - *creature*: The creature type is equivalent to the animal type, and keeping only animal is clearer.
    - *rabbit*: The rabbit type does not offer any different behavior than the animal type, and as such the rabbit type should be removed.
    - *relocation_strategy*: This is not a type, but the plan itself.
    - *hunger*: This is not a type, but a state.
    - *starting_positions*: This is better modeled with initial states.
    - *target_positions*: This is better modeled with goal states.
    - *conservatory*: There is only ever one conservatory, and as such it should not be included.
    - *path*: Paths are better modeled with a "path_between" predicate or similar.
```

## Example 2
### Domain and Problem
The AI agent is here a scheduling system for a house construction company with administrators who can create orders, workers who perform them, and vehicles to transport things.

Currently, there are a couple of trucks at the Chicago depot and we're trying to build three houses in the area. Jamie is the only administrator, but we've got Emma and Bob available for general work.

### Types
- location: Locations can be visited and traveled between.
- house: Constructed by the company. Are a type of location.
- scheduling_system: The scheduling system performing the planning.
- vehicle: Vehicles can transport things, such as material or personnel, between locations.
- truck: A type of vehicle for ground transport.
- jamie: An administrator.
- road: A road connecting two locations.
- emma: A general worker.
- build_roof: A roof is constructed.
- locality: The locality object describes where each worker is at each step.
- bob: A general worker.
- material: Assorted material for constructing houses. Transported by vehicles.
- city: The city in which construction takes place.

### Checklist
1: Are any types the exact same?
    *truck* and *vehicle* are similar, with *vehicle* being a parent type of *truck*. While there are differences in properties, they will have the same actions. Thereby one of them should be removed, and since *vehicle* is the more general and more commonly applicable type we keep this. So: Yes.

2: Are any of the types actually objects?
    *jamie*, *emma* and *bob* are all objects rather than types. These should be removed and replaced with the types *general_worker* and *administrator*. Hence: Yes.

3: Are any of the types actually actions?
    *build_roof* is an action, and not a type. Therefore it should be removed. So: Yes.

4: Are any of the types actually properties, relationships, connections, or otherwise better modeled using predicates (not counting `is_{type}` predicates)?
    The *road* property is better modeled with a predicate indicating a connection between locations. Therefore: Yes.

5: Is the acting agent itself or the resulting plans included?
    The *scheduling_system* is the AI agent itself. Additionally, this is not a physically embodied agent such as a robot (which are alright to model). Hence, this should not be modeled as a type. So: Yes.

6: Is it likely that there will only be a single instance of a given non-subtype in the current problem (not counting initial/goal states)?
    *location* will be used for all locations.

    *house* will be used for all three houses.

    The *scheduling_system* will only be used once and is not a subtype of anything else, but this has already been addressed above.

    *vehicle* will be used for all trucks.

    *truck* would be used for all trucks, but is addressed above.

    *road* would be used for all connections, but is addressed above.

    *build_roof* is addressed above.

    There will only be a single *locality* object and *locality* is not a subtype of anything else. As such this should be removed.

    *material* will be used for all materials.

    On the other hand, while it could be relevant to model several cities, the current problem only takes place in a single city. Hence, there will only be a single *city* instance and all houses will be in this. Therefore, *city* should not be kept.

    *general_worker* is used for Emma and Bob.

    There will however only be a single *administrator*. But, this is a type similar to *general_worker* and as such adding a *worker* metatype would make sense and make *administrator* not be alone. As such, it is still worthwhile to keep the *administrator* class due to being a specialized subtype with *worker* used for all workers. This is an example of when a type with only one instance can be kept, i.e. when it is used as a subtype of a more populated type (either novel or existing).

    As such: Yes.

7: Are any of the types modeling states (this could be initial states, goal states, or other large parts of states)?
    The *locality* type describes where the workers are, i.e. a large portion of the current state. As such, this should be removed. Neither the goal nor the initial state are in any way included. Hence: Yes.

8: Are any of the types unrelated to the domain being modeled?
    All types relate to the house construction domain. So: No.

### Feedback
I will now provide suggestions based exclusively, completely, and only according to the checklist. I will not suggest anything not identified by the checklist.
```
Replace the following types:
- *emma* and *bob* should be replaced with a *general_worker* type.
- *jamie* should be replaced with an *administrator* type.

Add the following types:
- *worker*. This is added as a joint parent type of "general_worker" and "administrator".

Remove the following types:
- *truck*: While trucks exist, their behavior will be the same as that of vehicles. As such, only vehicle should be kept.
- *build_roof*: This is an action and should not be included.
- *road*: This is better modeled with a predicate.
- *scheduling_system*: This is the AI agent itself and should not be included.
- *city*: There will only be a single city in the current problem, and it should therefore be excluded.
- *locality*: This describes a large subset of the state and should therefore be removed.
```

## Example 3
### Domain and Problem
I've got a building with five floors and two elevators, and I want you to plan the elevator trips for people in the building in a climate-friendly way.

### Types
- floor: The locations in which elevators and people move between.
- elevator: These move between floors and can carry people.
- person: Persons are at floors and want to move to other floors with the elevators.

### Checklist
1: Are any types the exact same?
    The included types are all distinct. So: No.

2: Are any of the types actually objects?
    The types are types. As such: No.

3: Are any of the types actually actions?
    The types aren't actions, they all reflect objects. Hence: No.

4: Are any of the types actually properties, relationships, connections, or otherwise better modeled using predicates (not counting `is_{type}` predicates)?
    The types are all concrete, and none describe properties. Therefore: No.

5: Is the acting agent itself or the resulting plans included?
    The agent is the elevator automator, and the plan is a sequence of elevator movements. Neither is included. So: No.

6: Is it likely that there will only be a single instance of a given non-subtype in the current problem (not counting initial/goal states)?
    *elevator* will be used for each of the elevators.

    *floor* will be used for all five floors.

    *person* will be used for all the people traveling.

    As such, each type is used several times. So: No.

7: Are any of the types modeling states (this could be initial states, goal states, or other large parts of states)?
    The state is described by the positions of elevators and people. The goal and initial state will also be such descriptions. *elevator* does not model such states. *floor* does not model such states. *person* does not model such states. Hence: No.

8: Are any of the types unrelated to the domain being modeled?
    The included types (floor, elevator, person) are all related to elevator transportation. So: No.

### Feedback
As the answer to each question was "No", I have no feedback to give. So: No feedback.
